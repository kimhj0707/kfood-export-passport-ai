# K-Food Export Passport AI

## 국가별 규정 데이터 검증 및 정리 지시서

**작성일**: 2026-01-16  
**목적**: 국가별 규정(`details` 데이터)이 실제 조사 근거에 기반한 것인지 검증하고,  
향후 반복 데이터 입력을 “더미가 아닌 도메인 데이터”로 고정하기 위함

---

## 1. 배경 및 문제 인식

현재 프로젝트에서는 알레르기 및 라벨 규정 경고 시  
아래와 같은 `details` 구조를 도입하여 **근거 기반 경고**를 제공하고 있다.

```json
{
  "risk": "Wheat allergen missing",
  "severity": "HIGH",
  "regulation": "FDA FALCPA",
  "article": "Section 403(w)",
  "reason": "Major allergen must be declared in plain English"
}
```

이 구조는 서비스의 신뢰도를 크게 향상시키는 방향이지만,
**국가별로 실제 조사·해석이 정확한지에 대한 검증이 필요**하다는 문제 제기가 있었다.

---

## 2. 핵심 질문 (검증 대상)

Gemini는 아래 질문에 답하는 것을 목표로 한다.

1. 현재 입력된 국가별 규정 근거는 실제 법·가이드라인에 기반한 것인가?
2. 국가별로 “안전한 표현”과 “위험한 단정 표현”은 무엇인가?
3. 반복 데이터 입력이 더미가 되지 않기 위해 필요한 최소 메타데이터는 무엇인가?

---

## 3. 국가별 1차 판단 요약 (기준선)

### 3-1. 미국 (US) – 비교적 안전 ✅

- `FDA FALCPA Section 403(w)`는 알레르기 표시의 핵심 조항으로 널리 인용됨
- “plain English로 food source name 표시” 취지는 FDA 가이던스와 일치
- 단, **알레르기 개수(8 vs 9)**는 법 개정(FASTER Act) 이후 혼동 가능성 존재

**판단**

- 조항 번호 + 규정명 + 취지 요약은 유지 가능
- 알레르기 개수는 UI/문서에서 단정 표현 주의 필요

---

### 3-2. 일본 (JP) – 숫자 단정 위험 ⚠️

- 자료에 따라:

  - 의무 7개 + 권장 21개 = 28개
  - 또는 총 27개로 설명되는 경우 존재

- 시기·문서·기관에 따라 표현이 달라짐

**판단**

- “27개 확정” 같은 숫자 단정은 위험
- mandatory / recommended 분리 표현 필요
- 기준 연도(as_of) 명시 필요

---

### 3-3. 베트남 (VN) – 규정 성격 주의 ⚠️

- Decree 43/2017/ND-CP는

  - 알레르기 특화 규정이라기보다
  - **수입 상품 포함 라벨 일반 표기 규정** 성격이 강함

**판단**

- “알레르기 의무 규정”으로 단정하면 부정확
- 라벨 필수 항목 / 언어 / 표기 방식 프레임으로 기술 필요

---

## 4. Gemini에게 요청할 작업 (명확 지시)

### 4-1. 검증 작업

각 국가별로 다음을 조사·정리할 것:

- 공식 규정/가이드 문서명
- 관할 기관
- 관련 조항(번호 또는 섹션명)
- 알레르기/라벨 관련 **정확한 규정 성격**
- 숫자/범위가 변동 가능한 경우 그 이유

---

### 4-2. 출력 형식 (필수)

Gemini의 최종 출력은 **아래 JSON 구조 예시를 따를 것**.

```json
"details": {
  "regulation": "공식 규정 또는 법령명",
  "article": "조항 번호 또는 섹션명",
  "reason": "해당 규정의 핵심 취지 요약 (단정 표현 금지)",
  "authority": "관할 기관",
  "as_of": "기준 연도 또는 문서 기준일",
  "note": "변동 가능성 또는 해석 주의사항 (있는 경우)"
}
```

---

## 5. 반복 데이터 입력의 성격 정의 (중요)

본 프로젝트에서 이후 진행되는 데이터 입력 작업은
새로운 로직 개발이 아닌,

> 이미 검증된 `details` 구조에
> 국가별·알레르기별 **실제 규정 지식을 채워 넣는
> Data Enrichment(도메인 지식 구조화) 작업**이다.

이는 더미 데이터가 아니며,

- 서비스 신뢰도
- 설명 가능성
- 공모전/발표 평가

에 직접적인 영향을 미치는 핵심 작업이다.

---

## 6. 금지 사항

- “추후 확인 필요”, “일반적으로 알려져 있음” 같은 모호한 표현 금지
- 출처 불명확한 블로그/2차 요약 자료 단독 인용 금지
- 숫자·의무 여부에 대한 단정적 서술 금지 (근거 없을 경우)

---

## 7. 최종 목표

- 국가별 규정 데이터가 **“의심 없이 설명 가능한 상태”**가 되는 것
- 심사위원/면접관 질문:

  > “이 근거는 어디서 나온 겁니까?”

에 대해 **한 문장으로 답변 가능한 수준** 확보

---

## 8. 한 줄 요약 (고정)

> 이 문서는 규정 데이터를 더미가 아닌
> **검증된 도메인 지식 자산으로 고정하기 위한 작업 지시서**다.
